# 汤鲜生：外部供应商对接与采购战略指南

针对“部分或大部分商品需要从第三方采购网站进货”的业务场景，我们需要在商业谈判、技术对接和系统架构上做前瞻性布局。

## 一、 核心业务模式确认（决定了技术怎么写）

在写代码之前，您必须先和这家供应商确认一个核心业务模式：货到底怎么发？

### 模式 A：集中采购（入库模式）

*   **逻辑**：您在他们网站下单 -> 他们把一大批货发到您攀枝花的 5000㎡ 冻库 -> 您再通过同城外卖/自提发给 C端/B端 客户。
*   **特点**：履约体验极好（依然能做到半小时达），但依然有囤货资金压力。

### 模式 B：一件代发 (Dropshipping 模式)

*   **逻辑**：客户在“汤鲜生”小程序下单 -> 系统自动在供应商网站下个相同的单 -> 供应商直接把货通过快递发给您的客户。
*   **特点**：零库存压力，无限扩展 SKU。但物流时效不可控，无法满足“同城半小时达”，只适合保质期长的预包装零食或干货底料。

---

## 二、 技术对接方案（按对方技术实力分级）

对方有“可采购网站”，这意味着我们可以通过技术手段把他们变成我们的“云端后备隐藏仓库”。

### 🥇 最佳方案：API 接口直连 (要求对方开放 API)

在合作前，第一件事就是问他们：“你们有没有开放平台（Open API）？”
如果他们有 API，那这是最完美的：

1.  **商品同步**：写一个 Vendure 定时任务（Cron Job），每天凌晨 3 点调用他们的 API，拉取最新商品、价格和库存，自动更新到您的数据库中。
2.  **自动推单**：客户在汤鲜生付款成功后，Vendure 触发 `OrderStateTransitionEvent`，直接通过 API 把订单推给供应商，实现 100% 自动化，无人值守。

### 🥈 中等方案：数据流导入/导出 (CSV / Excel)

如果对方是个相对传统的系统，没有 API，但后台能导出表格：

1.  **库存更新**：每天让对方运营发一份最新的库存/价格 Excel 表。我们在 Vendure 后台写一个批量导入插件，5 秒钟更新全站库存。
2.  **采购下单**：每天晚上 8 点，把当天汤鲜生收到的需要采购的订单，导出成一张标准 Excel 表，发给对方财务或录入对方网站。

### 🥉 极客方案：RPA 爬虫机器人 (对方系统极其封闭)

如果对方既没有 API，也不配合给表格，只能在他们网站上纯手工点：

*   我们可以在服务器上部署 RPA 脚本（如 Puppeteer / Playwright）。
*   相当于写一个“虚拟机器人”，模拟真人在浏览器里登录他们的网站，爬取商品价格自动同步给我们；有新订单时，机器人自动登录他们的网站，把商品加入购物车并结算。

*(注：此方案技术维护成本较高，仅作为对方极不配合时的兜底黑客手段。)*

## 三、 Vendure 后端的架构适配方案

在 Vendure 中处理这种“外部供应商”货源，架构上非常优雅，我们需要做以下两件事：

### 1. 引入 StockLocation (多仓库/虚拟仓库概念)

在 Vendure 中，不要把所有的库存都混在一起。

*   建立一个 **Location 1**: 攀枝花一号冻库（属于您自己的重资产，真实库存）。
*   建立一个 **Location 2**: 某某供应商云仓（这是虚拟的）。

当用户下单时，Vendure 会自动判断这个商品是从一号库扣减真实库存，还是仅仅扣减供应商的虚拟库存。

### 2. 利用 CustomFields 标记商品来源

我们需要在商品表（ProductVariant）里加两个自定义字段：

*   `supplierCode` (string): 供应商代码（如：SUP_A）。
*   `supplierCostPrice` (int): 供应商实时采购底价。

**作用**：这样系统能自动算出毛利，而且前端用户完全看不出来这个商品是我们自己库里的还是别人家代发的。

### 3. 开发“自动采购单生成器”插件 (采购助手)

为您的采购专员开发一个简单的 Vendure 内部插件：

1.  每天一键点击，系统自动把今天所有卖出的、带有 `supplierCode` 标记的商品汇总。
2.  生成一张清单：“今日需向 A 供应商补货：牛肉 50件，毛肚 20件”。
3.  拿着这张清单去他们的网站一键下单。在早期，这种**“半自动汇单 + 人工去对方网站下单”**是最稳妥、开发成本最低的方式。

## 四、 商业运营上的防坑建议

1.  **价格波动预警**：外部供应商如果突然涨价，而您的小程序没来得及改价，您就会亏钱。必须在系统中设定：如果采购成本大于售价的 80%，订单进入异常预警状态，暂停自动履约。
2.  **断货退款处理**：供应商网站上显示有货，您卖出去了，结果去他们那边采购时发现没货了。这就需要您的客服体系介入，及时给 C 端客户退款，并送上一张优惠券作为补偿。
3.  **商品白牌化**：如果您是从他们那里进“预包装零食”，可以直接卖；如果是进散装冻品，建议到您的冻库后，贴上“汤鲜生”的标签再发给客户，强化自身品牌，而不是做个无情的倒爷。

## 五、 多单位（袋/件）混合售卖与库存动态换算架构

针对供应商商品存在“1件10袋”、“1件20袋”等规格不一的复杂情况，绝对不能在代码中写死固定换算率。我们需要采用**“动态 SKU 换算”**模型。

### 1. Vendure 后端数据建模 (动态转化率)

我们利用 Vendure 的 `ProductVariant`（SKU）机制，为每一个规格创建独立的 SKU，并利用 `CustomFields` 绑定专属的动态换算系数。

在 `vendure-config.ts` 中新增自定义字段 `conversionRate`（代表该 SKU 包含多少个基础单位）：

*   **商品示例 A**：某品牌川粉（1件 = 10袋）
    *   Variant 1 (袋): 售价 4元, `conversionRate`: 1
    *   Variant 2 (件): 售价 38元, `conversionRate`: 10
*   **商品示例 B**：精品小酥肉（1件 = 20袋）
    *   Variant 1 (袋): 售价 15元, `conversionRate`: 1
    *   Variant 2 (件): 售价 280元, `conversionRate`: 20

**库存扣减拦截器**：写一个 Vendure Inventory Subscriber。当用户买走 2 件小酥肉时，拦截器读取其 `conversionRate`（20），执行 `2 * 20 = 40`。系统自动从该商品的“基础库存池”中精准扣减 40 袋的库存。

### 2. Taro.js 前端场景化 UI 分发

同一套底层数据，在前端根据用户身份（C端/B端）展示完全不同的点单交互：

*   **面向 C 端散客（侧重引导凑单）**：采用【底部弹窗选规格】
    *   列表页只展示起步价：“精品小酥肉 ￥15起”。
    *   用户点击 [+] 号，弹出 Bottom Sheet 规格选择器：`[ 一袋 ￥15 ] [ 整件装(20袋) ￥280 ]`
    *   利用“整件买更划算”的视觉对比，刺激 C 端家庭客囤货。

*   **面向 B 端餐饮老板（侧重极致效率）**：采用【列表平铺视图】
    *   B 端用户没时间点弹窗选规格，界面直接将其拆解为两行独立商品：
        *   📦 小酥肉 (袋) ￥15.00 [-] 0 [+]
        *   📦 小酥肉 (件/20袋) ￥280.00 [-] 3 [+]
    *   老板看着库房缺什么，直接狂点步进器，一键完成大宗扫货。

## 六、 商品数字化包装与精细化展示策略（针对外部货源的痛点解决）

引入外部供应商的商品时，往往会面临图片视觉不统一、分类不匹配、甚至带有第三方印记等问题。我们不仅要解决技术展示，更要建立**“零版权风险”**的品牌大一统视觉。

### 6.1 多级分类的无缝映射

*   **痛点**：供应商网站的分类可能是三级（冻品 -> 牛羊肉 -> 肥牛卷），我们需要在小程序里合理展示。
*   **解决方案**：Vendure 原生拥有极其强大的 Collections（集合）系统，天生支持无限层级的树状结构。
*   **执行**：在同步数据时，只需将供应商的类目名创建为 Vendure 嵌套的 Collections。前端 Taro.js 直接调用 collections GraphQL 接口，配合侧边栏（一级分类）和顶部 Tab（二级分类）即可完美渲染。

### 6.2 战略级白帽方案：AIGC 视觉重塑与品牌大一统（合规首选）

*   **商业逻辑与版权规避**：摒弃传统的“抹除/规避水印”等涉嫌灰产侵权的做法。我们不复刻第三方原图，不以去除水印为目的，而是基于合规授权，进行全新的数字化二次创作，彻底斩断版权风险。
*   **核心解法**：根据供应商授权的商品基础素材与规格信息，利用 AI 模型重新生成更统一、更美观的商品展示图。严格保留真实的商品属性、包装形态和必要描述文字，杜绝“货不对板”。

**技术实现路径 (AIGC Workflow)**：
1.  **特征解构与抠图**：从供应商授权的商品图中，提取出核心商品主体（如包装袋、底料块），剥离原本杂乱的背景或第三方印记。
2.  **场景重构 (Scene Generation)**：利用 Stable Diffusion (SD) 或商用电商 AI 生成接口（如阿里云/腾讯云商品图 API），将商品主体置入**“汤鲜生”专属的品牌视觉模版**中（例如：统一的深色暗调桌面、搭配火锅雾气与新鲜蔬菜点缀）。
3.  **信息保真 (ControlNet 强控)**：这是最关键的一环。通过引入 ControlNet 的 Canny（边缘检测）或 Lineart（线稿控制），将商品包装原本的几何轮廓、Logo 比例、净含量及中文字符死死锁住，防止 AI 重绘时产生“幻觉”捏造错误文字，确保 100% 的商品真实性还原。
4.  **核心生成指令 (System Prompting)**：将这一原则作为封装给 AI 绘图大模型的底层提示词（Prompt）规范：“根据授权商品素材与规格信息，生成更统一、更美观的商品展示图；保留真实商品属性、包装信息和必要描述文字，但不复刻第三方原图，不以去除或规避水印为目的。” 从而驱动整个自动化流水线完成高品质、零风险的视觉输出。

**降维优势**：零法律风险；让来自几十个不同供应商、原本风格五花八门（白底图、仓库实拍、纸箱图）的商品，瞬间统一成高级的“汤鲜生自营大片”风格，极大拉升 C 端食欲与客单价转化。

### 6.3 战术级备选：自动化瑕疵修复流水线（针对特定授权图的局部处理）

*   **适用场景**：对于部分供应商明确授权直接使用，但图片本身带有轻微杂乱背景、模糊干扰文字，且不适合用 AIGC 进行整体重绘的商品图（如纯 B 端进货参考图）。
*   **处理机制 (PaddleOCR + IOPaint)**：
    1.  **智能识别与定位**：不依赖死板的遮罩，引入 PaddleOCR 进行全图文字特征扫描。结合几何形态过滤（例如：识别并锁定倾斜角度为 25-35 度，长宽比 > 3:1 的干扰字符区域）。
    2.  **安全白名单防误伤**：设定图片中心区域免疫，并将包含“火锅”、“川粉”、“净含量”等核心商品描述属性的文字加入白名单，防止算法误伤正常商品信息。
    3.  **AI 局部填补**：利用本地部署的轻量级 CPU 模型 IOPaint (LaMa)，针对被标记的瑕疵 or 干扰区域进行物理级的无痕填补修复。

### 6.4 B/C 端商品图片隔离显示

*   **痛点**：有时候保留供应商的原始“大纸箱”实拍图，反而更能让 B端老板觉得“量大便宜、是批发价”；但 C端家庭主妇看纸箱会觉得没食欲，C端需要 AIGC 生成的精美摆盘图。
*   **解决方案**：利用 `CustomFields` 隔离图片
    *   在 `vendure-config.ts` 中，为 Product 实体添加一个自定义字段：`cEndFeaturedAsset`（类型为 relation，关联到 Asset 图片资源）。
    *   **后台操作**：默认的主图（`featuredAsset`）放供 B 端进货参考的原图；而在自定义的 `cEndFeaturedAsset` 中，上传 AIGC 重绘后的精美场景图。

**Taro 前端判定**：
```typescript
// 伪代码示例
if (user.isB2B) {
    render(product.featuredAsset); // B端老板看纸箱图，接地气
} else {
    // C端散客看 AIGC 精美重绘图，促冲动消费
    render(product.customFields.cEndFeaturedAsset || product.featuredAsset); 
}
```

### 6.5 B/C 端商品可见性严格限制

*   **痛点**：有些商用大桶底料（50斤装），C端散客根本买不了，展示给他们只会扰乱视线；同理，有些高溢价的小零食盲盒，B端老板觉得花里胡哨。
*   **解决方案**：可见性控制字段 (Visibility Flag)
    *   在 Product 中添加自定义枚举字段 `targetAudience`（目标客群），可选项为：`ALL` (通用), `B_ONLY` (仅限B端), `C_ONLY` (仅限C端)。
    *   **后端/前端过滤**：在 Taro.js 前端发起 GraphQL 查询商品列表时，将当前用户的身份作为参数传入。

**GraphQL 查询过滤示例**：
如果是未登录的散客或 C 端用户，只查询 `targetAudience` 不等于 `B_ONLY` 的商品，从而实现商品大盘的完美隐身与隔离。
